# Plan

## Context

This project starts from a simple observation:

[`pi-memory`](https://github.com/jayzeng/Pi-Memory) works, but its strongest behavior is optimized around process shutdown. That does not match my workflow well. I often:
- close pi quickly
- reopen it immediately with `pi --continue` or `pi --resume`
- use pi as a short-lived terminal tool instead of a long-lived app session

In that workflow, `session_shutdown` is not a reliable proxy for "I am done with this unit of work".

Pi already has better primitives than plain process lifecycle:
- persistent session files
- tree-based branching
- compaction hooks
- commands, tools, and UI hooks
- session resume and explicit session selection

The goal is to build memory around those primitives instead of fighting them.

## Core design decisions

### 1. Keep the default context extremely small

Pi is good because the default context is very small.

This project should preserve that. Memory must not turn every prompt into a large vault dump.

Implications:
- automatic injection must be tiny
- most retrieval should be dynamic, not automatic
- global memory should not be auto injected

### 2. No startup summarization step

Automatic memory loading must not require an extra LLM call on startup or before every prompt.

That means:
- no "current scratchpad summary" generated by a model at load time
- automatic scope loading should be pure file I/O
- a simple `tail -n N` style excerpt is preferred for the first design
- if summaries exist, they should be explicit and optional

### 3. Markdown first

The memory vault should be markdown first.

Use `.md` files everywhere by default. Only introduce JSON or JSONL later if a specific file becomes genuinely painful in markdown.

Reasons:
- human-readable
- Obsidian-friendly
- git-friendly
- easy to inspect with `rg`, `fd`, and standard shell tools
- simple enough that any TUI-focused developer can understand and work with it

The design target is compatibility, not special integration:
- the vault should be easy to traverse with whichever tools the user already prefers
- it may be friendly to tools like Obsidian or `qmd`
- but the project should not be designed around implementing special compatibility layers by default

### 4. Three scopes, plus inbox and daily trace

The system has three real memory scopes:

- **Private scope**
  - per session
  - private scratchpad and working notes
  - survives compaction, `/tree`, and reopening the same session
  - never leaks into parallel sessions
- **Slug scope**
  - per cwd or repo slug
  - centrally stored in the memory vault, not inside the repo
  - small durable repo memory across many sessions
  - overlaps with repo-local skills, but stays lighter and more agent-managed
- **Global scope**
  - flexible Zettelkasten-style durable note space
  - cross-session, cross-repo, and real-world memory

Additional buckets:
- **Inbox** for cheap capture through a future capture command or companion flow
- **Daily** as a tiny activity trace, not the main memory store

### 5. Cheap capture matters, but naming is still open

This project likely wants a low-friction capture primitive similar in spirit to org-mode capture.

The exact naming is still soft. Current placeholder names like `/in` and `/process-inbox` are not final decisions.

Important constraints for any future capture flow:
- it should exist both inside pi and outside pi as a shell command or script
- it should not require an LLM
- it should fundamentally be a file append operation with good conventions
- it may later live as a companion feature or even a separate extension

### 6. Daily files should stay lightweight

Daily files are useful as a small activity trace.

They should mostly answer:
- what happened today, in one line
- what durable note or workstream was touched

They are not expected to carry the full weight of long-lived memory.

### 7. Global scope is never auto injected

Global Zettelkasten notes should be accessed through explicit retrieval.

That means:
- no broad automatic global injection
- provide a skill or workflow for how to use global notes
- that skill can cover frontmatter conventions, note links, and `rg`

### 8. Optimize for driver UX, performance, and simple operation

This project is for developers driving an agent, not only for the LLM.

That means:
- user experience matters more than magic
- performance matters
- the system should stay understandable in plain files and shell tools
- the vault should remain usable by small local models and by humans browsing it directly
- avoid reimplementing heavyweight knowledge-base features when simple files and conventions are enough

This must not become slop.

## Scope model

### Private scope

Private scope is the session-local workspace.

What belongs here:
- temporary notes
- open questions
- near-term reminders
- session-specific context that should survive resume

The private scope belongs to the session, not the branch, at least for now.

This is an explicit ASAP simplicity choice.

### Slug scope

Slug scope is small repo memory.

What belongs here:
- repo-specific preferences
- recurring gotchas
- small local conventions
- notes that should survive many sessions in the same repo

Slug scope should stay in the central memory vault.

It should not live in the repo by default.

Important distinction from `AGENTS.md` and repo-local skills:
- `AGENTS.md` is published instruction context that pi loads as part of its context-file convention
- repo-local skills are curated capability or workflow artifacts
- slug scope is a lightweight draft layer for small repo memory and gotchas

A useful future flow is:
- capture small repo memory in slug scope first
- promote stable or important parts into `AGENTS.md` or repo-local skills later

### Global scope

Global scope is the durable note graph.

What belongs here:
- long-lived workstreams
- real-world tasks and reminders with lasting value
- personal preferences
- technical decisions with cross-repo impact
- notes that should remain useful beyond one cwd

This scope should use timestamp-prefixed Zettelkasten filenames like:

```text
20260423114819-someid.md
```

The filename is a stable ID plus a small human hint. The real title belongs in frontmatter and the note body.

## Storage layout

Current preferred layout:

```text
~/.pi/agent/native-memory/
  sessions/
    <slug>/
      <session-id>/
        scratchpad.md
        # checkpoints.md is reserved for later, not part of the current core plan

  slugs/
    <slug>/
      memory.md
      inbox.md

  notes/
    20260423114819-someid.md
    20260423121504-otherid.md

  daily/
    2026-04/
      2026-04-23.md
      2026-04-24.md

  inbox/
    inbox.md
```

Notes:
- `daily/` is sharded by year-month
- `sessions/` and maybe `slugs/` can be gitignored if desired because they are more machine-specific
- the whole directory should still work well as an Obsidian vault and as a git repo

## File semantics

### `sessions/<slug>/<session-id>/scratchpad.md`

Private session workspace.

This is the main automatically loaded scope.


### `slugs/<slug>/memory.md`

Small repo-scoped durable memory.

This is intentionally not the same thing as `AGENTS.md`.
It is a lightweight memory layer for gotchas and small repo context, not a published instruction file.

### `slugs/<slug>/inbox.md`

Repo-scoped inbox for candidate ideas that are not yet stable enough for `memory.md`.

### `notes/*.md`

Global durable Zettelkasten notes with YAML frontmatter.

Each note should have a stable opaque `id` in frontmatter.
The `id` should not be derived from the title.
A title can change without changing the note identity.

Current decision:
- use UUIDv4-derived note identities
- store them without dashes as 32 lowercase hex characters
- keep time ordering in the filename, not in the ID

This keeps the split clean:
- filename = timestamp ordering plus small human hint
- frontmatter `id` = stable opaque identity

### `daily/YYYY-MM/YYYY-MM-DD.md`

Tiny daily activity log.

### `inbox/inbox.md`

Global inbox for `/in` captures.

This is the cheap dumping ground for things that should not leak into current session context.

## Loading and retrieval model

Use a hot, lukewarm, and cold model.

### Hot memory

Automatically loaded:
- private scope only

Mechanism:
- no LLM summarization step
- use a small tail-style excerpt from `scratchpad.md`
- include the file path and omitted line count
- the agent can load more if needed

### Lukewarm memory

Possibly lightly loaded:
- slug scope

Mechanism:
- same primitive as private scope if auto-loading is enabled
- small tail-style excerpt only
- include path and omitted counts

Open point:
- whether slug scope should later gain optional tiny auto-loading or only a presence-only signal remains undecided

### Cold memory

Never broadly auto injected:
- global notes
- inbox files
- daily files
- full slug files

Access pattern:
- explicit read
- explicit search
- dedicated skill for global note usage

## Injection helpers

The system may later provide a command or tool that injects a summary of a specific scope into the current context.

Important distinction:
- this is optional and explicit
- it must not be part of the default load path
- it can exist so the agent can choose summary over full-file reading when useful

## Inbox processing

A future inbox-processing workflow should be interactive and collaborative, not automatic background cleanup.

The naming is still open.

The workflow should help turn inbox items into one of:
- private scratchpad content
- slug memory
- global note
- daily trace line
- discard or defer

## Session references

Some notes and scope files may store references to pi session files in frontmatter.

This is useful because `pi --session /path/to/session.jsonl` makes those paths operationally valuable.

Example shape:

```yaml
session_refs:
  - host: LECTOR-MBP-JK
    path: /Users/jkuball/.pi/agent/sessions/.../session.jsonl
```

Constraints:
- these references are host-specific
- they may go stale
- stale reference cleanup can be automated later
- the session file should not be copied into the vault

## Karpathy LLM wiki relation

This design is adjacent to the LLM wiki pattern:
- markdown vault as a maintained artifact
- Obsidian-friendly layout
- durable interlinked notes
- LLM-assisted maintenance over time

But this project adds pi-specific concerns that matter here:
- session-private workspace
- slug-scoped repo memory
- tiny default context
- shell-first capture through `/in`
- deliberate handling of resume, compaction, and session files

## Initial implementation plan

The project is still in Phase 0.

A major Phase 0 deliverable is to define a crisp Phase 1 and Phase 2 boundary so the first implementation stays small.

## Phase 0: design and examples

Goals:
- read pi extension, session, compaction, and package docs
- inspect example extensions relevant to state, compaction, and commands
- write down design constraints and storage semantics
- decide what is in Phase 1 versus Phase 2

Deliverables:
- README framing
- plan and storage docs
- initial vault layout
- initial loading model

## Proposed Phase 1 boundary

Phase 1 should only prove the core shape of the system.
It should be the smallest thing that demonstrates memory in the pi way.

Include in Phase 1:
- private session scratchpad in markdown
- slug memory skeleton in markdown
- global cold-storage note layout in markdown
- stable note identity conventions for global notes
- tiny automatic loading based on tail-style excerpts
- no startup summary LLM call
- no broad global injection
- enough commands or tools to inspect the private, slug, and global scopes directly
- basic manual creation and reading of global notes

Phase 1 does not need full link-resolution workflows.
It is enough if the agent can create notes and read existing files in a simple file-based way.

Phase 1 should also keep the memory-vs-instruction boundary clean:
- implement memory scopes
- do not blur them into `AGENTS.md` or repo-local skills yet
- leave promotion workflows to later phases

Explicitly keep out of Phase 1:
- inbox processing workflow
- capture command naming and implementation details
- automatic checkpointing around compaction
- promotion from slug memory into repo-local skills
- richer multimodal capture workflows beyond markdown links or file paths
- any dependency beyond what pi already ships with and what the user prefers to use externally
- backlink engines, sqlite indexes, or other attempts to reimplement a full PKM application

## Phase 1: markdown-first core memory prototype

Build the smallest useful version with the Phase 1 boundary above.

Success criteria:
- quick close and reopen with `pi --continue` retains useful private state
- parallel sessions do not bleed into each other
- global cold storage exists structurally and is usable manually
- default context stays very small
- vault remains easy to inspect in Obsidian, git, and shell tools

## Proposed Phase 2 boundary

Phase 2 should make the system pleasant to use.

Include in Phase 2:
- better driver-facing commands and workflows
- global Zettelkasten note creation and promotion flow
- daily trace updates
- optional session references in frontmatter
- cheap capture flow if it still proves valuable after Phase 1
- maybe a first explicit checkpoint command if it proves useful during Phase 1
- first pass on inbox items that reference local assets or other multimodal sources via markdown links or file paths

Still keep out of Phase 2 unless clearly needed:
- automatic compaction-driven checkpointing
- automatic slug-summary injection
- any special compatibility layer for external tools

## Phase 2: pleasant-to-use workflows

Add the Phase 2 items above.

Success criteria:
- captured items and notes can be processed collaboratively
- long-lived work moves into durable notes cleanly
- daily files stay short and useful
- driver UX improves without bloating the core system

## Phase 3: housekeeping and operability

Add:
- explicit and maybe automatic checkpoints in markdown if they prove worthwhile
- compaction-aware private scope preservation
- cleanup helpers for stale references and vault hygiene
- programmatic housekeeping tools that can run without LLM involvement
- better traversability for shell tools and small local models

Success criteria:
- compaction does not lose current workspace state
- housekeeping can be automated through shell scripts, cron, launchd, systemd, or similar tooling without forcing an LLM into the loop
- the vault stays readable and operable over time
- no housekeeping feature requires reimplementing a full note application database or backlink engine

## Phase 4: future experiments

This phase is intentionally vague.
It is the dumping ground for ideas once the core system is ready.

Possible examples:
- promotion from slug memory into `AGENTS.md` or repo-local skills
- companion extensions for capture or inbox workflows
- richer multimodal flows
- checkpoint ideas if a real need appears later
- future compatibility helpers for external tools if they ever prove worth it

## Current thesis

The extension should feel less like "global markdown notes with an exit hook" and more like this:
- a private session workspace
- a small slug-scoped repo memory
- a global markdown note graph
- a cheap capture inbox
- tiny automatic loading
- dynamic retrieval for everything else

That is the main bet of this project.
